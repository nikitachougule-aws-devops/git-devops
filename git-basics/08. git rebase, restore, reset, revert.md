# git rebase

Rebasing is the process of moving or reapplying commits from one branch onto another base branch.

---

## 🔹 Rebase vs Merge

### ✅ Rebase

- Moves feature branch commits on top of main  
- Creates linear history  
- No merge commit

### ✅ Merge

- Combines histories of two branches  
- Creates a merge commit  
- Keeps full history (non-linear)  

--


## git restore

- Restore working directory files (safe operation)

---

## git revert

- Undo a commit by creating a new commit

---

##  git reset

- Move the branch pointer and modify staging/working area  

### Types of Reset

#### 🟢 Soft Reset

- Keeps changes in staging area  
- Only removes commit  

👉 Use when: redo the last commit  

---

#### 🟡 Mixed Reset (Default)

- Unstages changes  
- Keeps them in working directory  

👉 Use when: rework changes before committing again  

---

#### 🔴 Hard Reset

- Deletes commits AND changes completely  

👉 Use when: completely discard work  

⚠️ Dangerous: data loss possible  

--- 

## 🔹 Key Difference

| Command | Affects History | Safe for Team |
|--------|----------------|---------------|
| restore | ❌ No | ✅ Yes |
| reset   | ✅ Yes (rewrites) | ❌ No |
| revert  | ❌ No (adds new commit) | ✅ Yes |

---

## 🔹 When to Use What

### ✅ Use restore

- Undo file changes before commit  

### ✅ Use reset

- Fix local commits  
- Clean up commit history before pushing  

### ✅ Use revert

- Undo changes in shared/public branches  

---

## 🔥 Real-World Scenarios

### Scenario 1

You edited a file incorrectly  
👉 Use `restore`

---

### Scenario 2

You made a bad commit locally  
👉 Use `reset` (soft/mixed)

---

### Scenario 3

You pushed a bug to production  
👉 Use `revert` (safe option)

---

## Golden Rule

- If the commit is already pushed → use **revert**  
- If it is local → you can use **reset**  

---

## 📌 Final Insight

- **restore** → file-level undo  
- **reset** → history rewrite (local use)  
- **revert** → safe undo for teams
